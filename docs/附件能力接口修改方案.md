# 附件与内嵌图片接口修改方案

## 1. 修改背景

原接口文档中，通用请求格式默认是 JSON，并明确写了“当前版本不提供附件上传接口，不涉及 `multipart/form-data`”；同时“当前版本不做的接口和能力”里也把“附件上传”列为不纳入范围。

现在为了支持邮件附件和正文内嵌图片，需要在原接口文档基础上做一次最小改造：

- 新增通用上传文件接口。
- 新增通用下载文件接口。
- 发送邮件接口支持一个附件。
- 邮件详情接口返回附件信息。
- 邮件正文 `RichTextNode.image.resourceId` 可以引用已上传的图片文件。

## 2. 总体方案

本次仍然坚持最小可落地版本：

```text
一封邮件最多支持 1 个附件
一封邮件可以包含多张正文内嵌图片
附件和内嵌图片都先上传文件，得到 fileId
发送邮件时携带 attachmentFileId
发送邮件时正文 image 节点携带 resourceId
查看邮件详情时返回 attachment
下载文件时校验当前用户是否有权访问该文件
```

附件和内嵌图片的区别：

| 类型 | 字段位置 | 用途 | 当前限制 |
|---|---|---|---|
| 附件 | `SendMailRequest.attachmentFileId` / `MailDetailVO.attachment` | 邮件底部附件卡片，可下载 | 最多 1 个 |
| 内嵌图片 | `RichTextNode.resourceId` | 邮件正文内展示图片 | 可多张，仅允许图片类型 |

`resourceId` 和 `fileId` 使用同一套文件 ID。为了兼容原 `RichTextNode` 设计，正文图片节点仍使用字段名 `resourceId`，但它的值就是上传接口返回的 `fileId`。

## 3. 新增接口

### 3.1 上传文件

```http
POST /api/files
```

请求格式：

```http
Content-Type: multipart/form-data
Authorization: Bearer <token>
```

请求参数：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| file | file | 是 | 上传文件 |

支持文件类型：

```text
.png / .jpg / .jpeg / .pdf / .docx / .zip
```

用途限制：

- 附件可以使用以上全部支持类型。
- 内嵌图片只能使用 `.png / .jpg / .jpeg`。
- `.pdf / .docx / .zip` 只能作为附件，不能作为正文图片。

建议限制：

```text
单个文件最大 10MB
```

成功响应采用最小结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "fileId": "file_20260603_001"
  }
}
```

失败情况：

| 场景 | code | message |
|---|---:|---|
| 未登录 | 40002 | 请先登录 |
| 文件为空 | 40000 | 文件不能为空 |
| 文件类型不支持 | 40000 | 文件类型不支持 |
| 文件过大 | 40000 | 文件大小不能超过 10MB |
| 上传失败 | 50000 | 文件上传失败 |

### 3.2 下载文件

```http
GET /api/files/{fileId}/download
```

请求头：

```http
Authorization: Bearer <token>
```

路径参数：

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| fileId | string | 是 | 文件 ID |

成功时返回文件流，不走统一 JSON：

```http
Content-Type: application/pdf
Content-Disposition: attachment; filename*=UTF-8''实验报告.pdf
```

说明：

- 附件下载建议使用 `Content-Disposition: attachment`。
- 正文图片渲染可以复用该接口，但前端不能直接依赖 `<img src>` 自动携带 token。
- MVP 前端应通过 `fetch/axios` 携带 Bearer Token 获取 Blob，再转成 `URL.createObjectURL(blob)` 渲染图片。
- 后续如需优化，可以新增短期签名 URL，不放入本次最小版本。

错误时仍然返回统一 JSON：

```json
{
  "code": 40003,
  "message": "无权限下载该文件",
  "data": null
}
```

下载权限规则：

```text
1. 未登录不能下载。
2. 文件不存在，返回 40004 文件不存在。
3. 文件未绑定邮件时，只允许上传者本人访问。
4. 文件绑定邮件后，只有该邮件的发件人和收件人可以下载。
5. 不允许通过真实文件路径直接下载，只能通过 fileId 下载。
```

## 4. 修改发送邮件接口

原来的 `POST /api/mails` 请求体只有：

```text
recipientUsername
subject
content
```

现在新增一个非必填字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| attachmentFileId | string | 否 | 附件文件 ID；当前版本一封邮件最多一个附件 |

正文内嵌图片不新增独立字段，继续使用 `content`：

```json
{
  "type": "image",
  "resourceId": "file_20260603_002"
}
```

修改后请求示例：

```json
{
  "recipientUsername": "zhangsan",
  "subject": "实验报告提交提醒",
  "content": [
    {
      "type": "paragraph",
      "children": [
        {
          "type": "text",
          "text": "您好，请注意本周五之前提交实验报告。"
        }
      ]
    },
    {
      "type": "image",
      "resourceId": "file_20260603_002"
    }
  ],
  "attachmentFileId": "file_20260603_001"
}
```

发送邮件时新增校验：

```text
1. attachmentFileId 不传：正常发送无附件邮件。
2. attachmentFileId 传了：文件必须存在。
3. attachmentFileId 对应文件必须由当前登录用户上传。
4. attachmentFileId 对应文件不能已经绑定到其他邮件。
5. attachmentFileId 对应文件可以是支持列表内任意类型。
6. content 中每个 image.resourceId 都必须存在。
7. image.resourceId 对应文件必须由当前登录用户上传。
8. image.resourceId 对应文件不能已经绑定到其他邮件。
9. image.resourceId 对应文件必须是图片类型。
10. 校验通过后，将附件和正文图片都绑定到当前邮件。
```

新增失败情况：

| 场景 | code | message |
|---|---:|---|
| 文件不存在 | 40004 | 文件不存在 |
| 使用了别人的文件 | 40003 | 无权限使用该文件 |
| 文件已绑定其他邮件 | 40000 | 文件已被使用 |
| 非图片文件作为内嵌图片 | 40000 | 内嵌图片文件类型不支持 |

## 5. 修改邮件详情接口

在 `GET /api/mails/{mailId}` 的详情响应中新增：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| attachment | object/null | 是 | 当前邮件附件；无附件时为 null |

有附件时示例：

```json
{
  "attachment": {
    "fileId": "file_20260603_001",
    "originalFilename": "实验报告.pdf",
    "contentType": "application/pdf",
    "fileSize": 204800,
    "downloadUrl": "/api/files/file_20260603_001/download"
  }
}
```

无附件时：

```json
{
  "attachment": null
}
```

正文内嵌图片仍然包含在 `content` 里：

```json
{
  "type": "image",
  "resourceId": "file_20260603_002"
}
```

前端渲染详情时：

- `attachment` 用附件卡片展示。
- `content` 中的 `image.resourceId` 通过 `/api/files/{fileId}/download` 拉取 Blob 后渲染。
- 如果图片文件下载失败，该图片节点显示加载失败占位，不影响邮件正文其他内容展示。

## 6. 数据库最小修改

本次建议新增文件资源表和邮件文件关系表。不要只在邮件表上增加 `attachment_file_id`，因为正文内嵌图片可能有多张，后续也可能扩展多附件。

### 6.1 新增文件资源表

```sql
CREATE TABLE file_resource (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  file_id VARCHAR(64) NOT NULL UNIQUE COMMENT '对外暴露的文件ID',
  uploader_id BIGINT NOT NULL COMMENT '上传用户ID',
  original_filename VARCHAR(255) NOT NULL COMMENT '原始文件名',
  stored_filename VARCHAR(255) NOT NULL COMMENT '服务端保存文件名',
  storage_path VARCHAR(500) NOT NULL COMMENT '服务端存储路径',
  content_type VARCHAR(100) NOT NULL COMMENT 'MIME类型',
  file_ext VARCHAR(20) NOT NULL COMMENT '文件扩展名',
  file_size BIGINT NOT NULL COMMENT '文件大小，单位byte',
  status VARCHAR(20) NOT NULL DEFAULT 'TEMP' COMMENT 'TEMP/BOUND',
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

### 6.2 新增邮件文件关系表

```sql
CREATE TABLE mail_file_relation (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  mail_id BIGINT NOT NULL COMMENT '邮件ID',
  file_id VARCHAR(64) NOT NULL COMMENT '文件ID',
  usage_type VARCHAR(20) NOT NULL COMMENT 'ATTACHMENT/INLINE_IMAGE',
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY uk_mail_file (mail_id, file_id),
  KEY idx_file_id (file_id),
  KEY idx_mail_usage (mail_id, usage_type)
);
```

约束建议：

- 当前版本后端业务层保证一个 `mail_id` 最多一条 `usage_type = ATTACHMENT`。
- `usage_type = INLINE_IMAGE` 可以多条。
- 文件发送绑定成功后，`file_resource.status` 从 `TEMP` 改为 `BOUND`。
- 关系表中出现的 `file_id` 必须来自 `file_resource.file_id`。

## 7. 当前仍然不做的内容

这次只是补最小文件能力，不做复杂文件系统：

```text
不做多附件
不做附件在线预览
不做附件删除
不做断点续传
不做文件夹管理
不做病毒查杀
不做 .7z
不做公开文件访问
不做短期签名 URL
不做图片压缩和裁剪
```

原文档“当前版本不做的接口和能力”中应把“附件上传”改成：

```text
复杂文件能力：当前版本不做。
当前只支持单附件上传、单附件下载、邮件附件展示和正文内嵌图片展示。
```

## 8. 最终接口清单变化

新增：

| 模块 | 方法 | 接口 | 优先级 | 说明 |
|---|---|---|---|---|
| 文件 | POST | /api/files | P1 | 上传文件，返回 fileId |
| 文件 | GET | /api/files/{fileId}/download | P1 | 下载文件，需校验权限 |

修改：

| 接口 | 修改内容 |
|---|---|
| POST /api/mails | 请求体新增 `attachmentFileId`，非必填；`content` 中允许 `image.resourceId` 引用图片文件 |
| GET /api/mails/{mailId} | 响应体新增 `attachment`，无附件时为 null；`content` 中的图片节点保持 `resourceId` |
| 当前版本不做的接口和能力 | 删除“附件上传完全不做”的表述，改为“不做复杂文件能力” |

## 9. 推荐结论

本次最小修改方案就是：

```text
新增上传文件接口；
新增下载文件接口；
发送邮件时可携带 attachmentFileId；
正文内嵌图片通过 RichTextNode.image.resourceId 引用 fileId；
邮件详情返回 attachment；
当前版本一封邮件最多一个附件；
当前版本允许多张正文内嵌图片；
下载时必须校验当前用户是邮件发件人或收件人，或文件未绑定时的上传者本人。
```

这个方案把“附件”和“正文图片”统一为文件资源，但在邮件接口层保持清晰边界。它不会破坏现有 `content: RichTextNode[]` 设计，也为后续多附件、图片预览和签名 URL 留出扩展空间。
